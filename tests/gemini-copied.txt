2. Standard Lego Node Implementation
If you are writing a custom Python or C++ node to control a basic Lego mechanism (like an actuator, claw, or simple motor), you will use the rclpy (ROS 2 Python) library. [1, 2]
A basic Python node setup to publish or subscribe to your Lego brick actions looks like this:

python
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class LegoBrickNode(Node):

    def __init__(self):
        super().__init__('lego_brick_node')
        self.publisher_ = self.create_publisher(String, 'lego_status', 10)
        timer_period = 0.5  # seconds
        self.timer = self.create_timer(timer_period, self.timer_callback)
        self.i = 0

    def timer_callback(self):
        msg = String()
        msg.data = f'Brick State: {self.i}'
        self.publisher_.publish(msg)
        self.get_logger().info(f'Publishing: "{msg.data}"')
        self.i += 1

def main(args=None):
    rclpy.init(args=args)
    lego_node = LegoBrickNode()
    rclpy.spin(lego_node)
    lego_node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

Yes, there are active community examples and clear pathways for this. [1]
Sets like the Mould King 15075 (AlphaDog / Quadruped) or chain/track-based excavators use multiple Power Functions-compatible motors (typically 2 to 4 channels) managed via an integrated Bluetooth battery hub rather than traditional individual wires. [1, 2]
To control these multi-axis systems using ROS 2 on a Raspberry Pi, choose one of two distinct integration architectures depending on your hardware preference.



Method 1: The Wireless Bluetooth Approach (No Extra Hardware)
Instead of wiring physical cables, the Raspberry Pi can control the Mould King battery hub over Bluetooth Low Energy (BLE). You map ROS 2 Twist messages or joint velocities directly to BLE commands. [1]
The Blueprint: Use the open-source mkconnect-python library, which is specifically reverse-engineered to control Mould King 4.0 / 6.0 smart hubs over Bluetooth. [1]
ROS 2 Architecture: You create a standard ROS 2 node that imports mkconnect. The node subscribes to a command topic (like /cmd_vel or /joint_states) and translates those numbers into multi-axis motor percentages.

Python ROS 2 Node Example (Wireless 3-Axis)

python
import rclpy
from rclpy.node import Node
from geometry_coordinate_msgs.msg import Vector3 # Using Vector3 to represent X, Y, Z axes
# Make sure to pip install mkconnect on your Pi before running
from mkconnect import mkconnect, hubcmd 

class MouldKingMultiAxisNode(Node):
    def __init__(self):
        super().__init__('mould_king_axis_node')
        
        # Initialize connection to Mould King Hub
        self.get_logger().info('Connecting to Mould King Hub via Bluetooth...')
        mkconnect() 
        
        # Subscribe to a 3-axis control topic
        self.subscription = self.create_subscription(
            Vector3,
            'axis_commands',
            self.listener_callback,
            10)

    def listener_callback(self, msg):
        # Map X, Y, Z float values (-1.0 to 1.0) to Mould King motor speeds (-100 to 100)
        speed_axis_A = int(msg.x * 100)
        speed_axis_B = int(msg.y * 100)
        speed_axis_C = int(msg.z * 100)
        
        # Send commands directly to Ports A, B, and C on the hub
        hubcmd(0, port='A', speed=speed_axis_A)
        hubcmd(0, port='B', speed=speed_axis_B)
        hubcmd(0, port='C', speed=speed_axis_C)
        
        self.get_logger().info(f'Sent Axis Speeds -> A: {speed_axis_A}, B: {speed_axis_B}, C: {speed_axis_C}')

def main(args=None):
    rclpy.init(args=args)
    node = MouldKingMultiAxisNode()
    rclpy.spin(node)
    rclpy.shutdown()

Method 2: The Wired Hardware Approach (For Precise Control)
If you require closed-loop control (reading encoders to know exactly how far the legs or tracks have moved), the wireless hubs fall short because they lack data feedback. For precise multi-axis kinematics (like walking gaits), you must bypass the Mould King hub and wire the motors directly to the Pi. [1]
The Blueprint: Use a Raspberry Pi Build HAT or a Mecanum/DC Motor HAT stacked directly on top of your Pi.
ROS 2 Architecture: Leverage the official ros2_control framework, utilizing its diff_drive_controller(for chain/track sets) or a custom PositionJointInterface (for the leg linkages of the 15075 Dog). [1, 2, 3, 4]

Direct Analogy: The LEGO Audi / Spike Framework
A highly relevant open-source project is the ROS2 Control of the LEGO Audi, which utilizes a Raspberry Pi paired with a Build HAT to manage multi-axis movements (steering and driving) via standard ROS 2 control loops. [1, 2]
To implement this wired approach:
Cut or adapt the Mould King motor cables to standard pins, or use a HAT with Lego-compatible ports.
Configure a ros2_control hardware description file (.xacro) to map your physical axes: [1]

xml
<ros2_control name="MouldKingHardwareInterface" type="system">
  <hardware>
    <plugin>mould_king_hardware/MouldKingSystemHardware</plugin>
  </hardware>
  <joint name="axis_1_joint">
    <command_interface name="velocity"/>
    <state_interface name="position"/>
  </joint>
  <joint name="axis_2_joint">
    <command_interface name="velocity"/>
    <state_interface name="position"/>
  </joint>
</ros2_control>

Which should you choose?
Choose Method 1 (Wireless) if you want to keep the Mould King model exactly as built out of the box and just want to map a keyboard, joystick, or navigation node to make it move. [1, 2]
Choose Method 2 (Wired) if you want to implement advanced robotics concepts like inverse kinematics, SLAM, or precise trajectory planning where missing steps or slipping causes failures. [1]

Moving from a 3-axis land platform (like a quadruped or a tracked excavator) to an aquatic boat requires an architecture that is highly modular, water-resistant, and scalable.
Because Lego-style Bluetooth hubs are not waterproof and lack precise data feedback, you should opt for standard hobby-grade electronics controlled by a Raspberry Pi. This allows you to easily swap land-based actuators for marine thrusters while keeping the exact same ROS 2 codebase.



The Recommended Setup: Standard PWM Controllers
Instead of buying proprietary brick hubs, use standard DC Brushed Motors (with Lego-compatible axle housings) or Hobby Servos.
You control them using a single hardware expansion board attached to your Raspberry Pi: the PCA9685 16-Channel PWM HAT.

Why this works for both Land and Water:
For Land (Mould King / Tracks): Ports 0, 1, and 2 send signals to motor drivers (like the L298N or Cytron MD10C) to control the 3 geometric axes.
For Sea (Boat): You unplug the land motors and plug a Marine Electronic Speed Controller (ESC) into Port 0 (for the propeller) and a standard Waterproof Servo into Port 1 (for the rudder).
Software: The ROS 2 node remains virtually identical. It simply publishes PWM values to specific pins.



ROS 2 Node Architecture for Modular 3-Axis Control
This Python node uses a generic architecture. It reads an incoming 3-axis command (X, Y, Z) and translates it into PWM signals.
To transition from the land robot to a boat, you do not rewrite this node; you simply change the wiring connections or adjust the scale factors in a parameter file.

python
import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Vector3
# Requires pip install adafruit-circuitpython-pca9685
import board
import busio
from adafruit_pca9685 import PCA9685

class ModularBrickNode(Node):
    def __init__(self):
        super().__init__('modular_brick_node')
        
        # Initialize I2C and PCA9685 Hardware
        i2c = busio.I2C(board.SCL, board.SDA)
        self.pca = PCA9685(i2c)
        self.pca.frequency = 50 # Standard frequency for RC servos and ESCs
        
        # Map axes to physical PCA9685 channels
        self.CHANNEL_AXIS_1 = 0  # Land: Left Track  | Boat: Main Thruster (ESC)
        self.CHANNEL_AXIS_2 = 1  # Land: Right Track | Boat: Steering Rudder
        self.CHANNEL_AXIS_3 = 2  # Land: Arm/Boom     | Boat: Unused (or Camera Tilt)

        self.subscription = self.create_subscription(
            Vector3,
            'cmd_axes',
            self.axes_callback,
            10)
        self.get_logger().info('Modular ROS 2 Brick Node Started.')

    def axes_callback(self, msg):
        # Convert incoming values (-1.0 to 1.0) into PWM duty cycles (0 to 65535)
        # 15% to 85% duty cycle is typical for hobby motor drivers/servos
        pwm_1 = int(((msg.x + 1.0) / 2.0) * (55700 - 9800) + 9800)
        pwm_2 = int(((msg.y + 1.0) / 2.0) * (55700 - 9800) + 9800)
        pwm_3 = int(((msg.z + 1.0) / 2.0) * (55700 - 9800) + 9800)

        # Write directly to the hardware pins
        self.pca.channels[self.CHANNEL_AXIS_1].duty_cycle = pwm_1
        self.pca.channels[self.CHANNEL_AXIS_2].duty_cycle = pwm_2
        self.pca.channels[self.CHANNEL_AXIS_3].duty_cycle = pwm_3

def main(args=None):
    rclpy.init(args=args)
    node = ModularBrickNode()
    rclpy.spin(node)
    node.pca.deinit()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

Designing for the Boat Migration
If you plan to transition to a boat later, keep these design considerations in mind today:
Keep Electronics Centralized: Build your Raspberry Pi, PCA9685 HAT, and battery system into a single, removable Lego "electronics pod." When switching hulls, you can lift the entire pod out as one piece.
Standardize Signal Types: By using standard Servo/PWM signals for your land robot today, transitioning to a boat later is a drop-in replacement. A marine ESC accepts the exact same PWM signals as a standard hobby servo.
Plan for Waterproofing: Lego bricks are not watertight. For the boat phase, plan to place your electronics pod inside a sealed plastic container (like a Tupperware tub) mounted onto a Lego catamaran hull, running long drive shafts out to the water.
